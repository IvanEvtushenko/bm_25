# Анатомия BM25-индекса

Здесь описано, как устроен индекс по корпусу текстов: какие файлы лежат на диске, какие структуры в памяти, как они связаны и что происходит при сборке и поиске.

Это документ «для возвращения через неделю». Если что-то непонятно — пишите дополнения сюда же.

## 1. Концепт: что такое BM25-индекс

BM25 — формула, которая для **одного запроса** и **одного документа** возвращает число «релевантность». Чтобы быстро отвечать на запросы по корпусу из тысяч документов, нужно заранее посчитать и сохранить:

1. **Словарь** — все уникальные «термины» (леммы), встречающиеся в корпусе.
2. **Матрицу частот** `tf[i, j]` — сколько раз термин `j` встречается в документе `i`.
3. **Длины документов** в токенах — для нормализации в формуле BM25.
4. **Document frequency** `df[j]` — в скольких документах встречается термин `j`.

Этого хватает, чтобы за один матрично-векторный проход посчитать BM25-скор всех документов для любого запроса. Никакой нейросети, никаких эмбеддингов — алгоритм 1994 года.

Доп. структуры в нашем проекте:
- `doc_ids[i]` — какой документ лежит в строке `i` матрицы `tf`
- `kinds[i]`, `pub_ids[i]` — тип и принадлежность документа (для фильтров типа «только элементы», «не из этой публикации»)
- `offsets[i]` — байтовая позиция документа в `docs.jsonl` (для lazy-load)

## 2. Источник правды vs производные данные

Это важное архитектурное разделение. На диске два класса файлов:

```
data/
├── docs.jsonl            ← ИСТОЧНИК ПРАВДЫ. Append-only.
│                          Одна публикация/группа элементов = одна строка JSON.
│                          Все остальные данные индекса — производные от него.
│
└── index/                ← ПРОИЗВОДНЫЙ КЭШ. Пересобирается из docs.jsonl.
    ├── tf.npz            ← scipy.sparse CSR матрица [N × V]
    ├── doc_lens.npy      ← np.int32[N]
    ├── df.npy            ← np.int32[V]
    ├── offsets.npy       ← np.int64[N]
    ├── vocab.json        ← {лемма: col_index}
    ├── doc_ids.json      ← list[str], len = N
    ├── kinds.json        ← list[str], len = N
    ├── pub_ids.json      ← list[str], len = N
    └── meta.json         ← диагностика: версия, k1, b, avgdl, и т.д.
```

Если `data/index/` удалить — это не страшно, [add_doc.py](../add_doc.py) пересоберёт его из `docs.jsonl`. Если удалить `docs.jsonl` — потеряли данные.

## 3. Ключ к навигации: параллельные массивы по `row_index`

Это самая важная идея в устройстве индекса. **`row_index`** — это позиция документа в матрице `tf`. Все массивы длины N — параллельны по этому индексу.

```
row_index = 0
   ├── tf[0, :]        — частоты лемм для документа 0
   ├── doc_lens[0]     — длина документа 0 в токенах
   ├── doc_ids[0]      — "pub:Аналитика/22"
   ├── kinds[0]        — "publication"
   ├── pub_ids[0]      — "Аналитика/22"
   └── offsets[0]      — байт N в docs.jsonl, где лежит этот документ

row_index = 1
   ├── tf[1, :]
   ├── doc_lens[1]
   ├── doc_ids[1]      — "pub:Аналитика/18"
   ├── kinds[1]        — "publication"
   ├── pub_ids[1]      — "Аналитика/18"
   └── offsets[1]
   ...
```

Аналогично по `col_index` — все массивы длины V параллельны:

```
col_index = 42
   ├── vocab[<term>] = 42      — обратное отображение из лемма "инфляция"
   ├── df[42]                  — в скольких документах встречается лемма
   └── tf[:, 42]               — её частоты по всем документам (колонка)
```

Формула связи: чтобы найти `tf` (term frequency) леммы `"инфляция"` в документе `"pub:Аналитика/22"` — две таблицы поиска:

```python
row = doc_ids.index("pub:Аналитика/22")    # обратная находится через _id_to_row в LazyDocuments
col = vocab["инфляция"]
freq = tf[row, col]
```

## 4. Как устроена матрица `tf` (scipy.sparse CSR)

`tf` имеет форму `[N × V]`. На текущих данных это `3 606 × 4 612 ≈ 16.6M ячеек`. Из них ненулевых — около `200k` (≈ 1%). Хранить плотно — 60 МБ, разреженно — единицы МБ.

**CSR (Compressed Sparse Row)** хранит три массива:
```
data    = [3, 1, 2, 5, 1, ...]   все ненулевые значения подряд (по строкам)
indices = [42, 17, 88, 42, 3, ...] какие у них столбцы
indptr  = [0, 3, 5, 8, ...]      где начинается каждая строка в data/indices
```

Элементы строки `i` лежат в `data[indptr[i]:indptr[i+1]]`, их столбцы — в `indices[indptr[i]:indptr[i+1]]`.

При сборке мы заполняем построчно (один документ — одна строка), поэтому CSR удобен. **При поиске** же мы наоборот идём по столбцам (по терминам запроса), и тут CSR медленный — нужен CSC. В [bm25/index.py:tf_csc](../bm25/index.py) есть ленивая конвертация: при первом обращении к `index.tf_csc` создаётся CSC-копия и кэшируется.

## 5. Что лежит в каждом файле индекса

### `tf.npz` — разреженная матрица частот

`scipy.sparse.save_npz` сохраняет CSR как zip-архив из трёх numpy-массивов (`data`, `indices`, `indptr`). При загрузке через `load_npz().tocsr()` получаем готовую CSR обратно.

### `doc_lens.npy` — длины документов в токенах

`np.int32[N]`. Нужно для нормализатора BM25:
```
norm[i] = 1 − b + b · doc_lens[i] / avgdl
```
Длинные документы получают `norm > 1`, что увеличивает знаменатель в формуле BM25 → их скоры ниже. Это и есть «штраф за длину».

### `df.npy` — document frequency

`np.int32[V]`. `df[j]` = в скольких документах встречается термин `j`. Из неё считается IDF:
```
idf[j] = log((N − df[j] + 0.5) / (df[j] + 0.5) + 1)
```

IDF на диск **не сохраняется** — пересчитывается при загрузке из `df` и `N`. Так автоматически поддерживается консистентность (если когда-то изменится N — IDF не «уплывёт»).

### `offsets.npy` — позиции строк в `docs.jsonl`

`np.int64[N]`. `offsets[i]` = байтовая позиция начала строки i-го документа в `docs.jsonl`. Используется в `LazyDocuments` для ленивого чтения:
```python
file.seek(offsets[row])
line = file.readline()
doc = json.loads(line)
```

Размер: для 1.7M документов это 14 МБ — копейки по сравнению с 3–5 ГБ, которые занял бы весь `docs.jsonl` загруженный в `dict` в Python.

### `vocab.json` — лемма → индекс столбца

Обычный dict, сериализованный в JSON. Например:
```json
{"инфляция": 42, "ожидание": 17, "облигация": 88, ...}
```

При запросе мы токенизируем запрос → получаем леммы → находим их `col_index` через `vocab[term]`. OOV (термины, которых нет в vocab) тихо пропускаются.

### `doc_ids.json`, `kinds.json`, `pub_ids.json` — метаданные документов

Три параллельных списка длины N. Доступны напрямую как `index.doc_ids`, `index.kinds`, `index.pub_ids`.

Зачем хранить отдельно от документов в JSONL: чтобы при поиске не лезть в JSONL ради простых фильтров. Например, в [coverage.py](../coverage.py) маска «элементы из других публикаций» строится из `index.pub_ids` и `index.kinds` без чтения с диска вообще.

### `meta.json` — диагностика

Читаемый JSON с параметрами индекса:
```json
{
  "version": 4,                  ← INDEX_VERSION, при несовпадении load_index выдаст ошибку
  "n_docs": 3606,
  "vocab_size": 4612,
  "avgdl": 59.93,                ← средняя длина документа (для BM25)
  "k1": 1.5,                     ← гиперпараметр насыщения tf
  "b": 0.75,                     ← гиперпараметр штрафа за длину
  "fields_by_kind": {            ← снимок INDEX_FIELDS_BY_KIND для отладки
    "publication": ["name", "business_description", "keywords"],
    "element": ["report_name", "name", "description"]
  },
  "kind_counts": {"element": 3321, "publication": 285}
}
```

При загрузке индекса первым делом проверяется `version`. Если она ниже текущей — ошибка с просьбой пересобрать индекс. Это страхует от загрузки старого формата, где не было, например, `offsets.npy` или `pub_ids.json`.

## 6. Как устроен `docs.jsonl` (источник правды)

Одна строка — один JSON-объект (документ). Никаких вложенных структур, всё плоско. Пример строк:

```json
{"pub_id": "Аналитика/22", "name": "Кредит экономике...", "business_description": "...", "doc_id": "pub:Аналитика/22", "kind": "publication"}
{"pub_id": "Аналитика/22", "name": "Дата | Тип жилья | Год | ...", "description": "...", "element_names": ["Дата", "Тип жилья", "Год", ...], "n_elements": 5, "doc_id": "el:Аналитика/22#g0", "kind": "element"}
```

Каждый документ обязательно имеет:
- `doc_id` — глобально уникальный идентификатор (`pub:<pub_id>` для публикаций, `el:<pub_id>#g<group_idx>` для групп элементов)
- `kind` — `"publication"` или `"element"`
- `pub_id` — связь с публикацией (для элементов это родительская публикация)

Остальные поля зависят от типа: для публикации это `name`, `business_description`, `keywords`, ...; для группы элементов — `name` (склейка имён элементов через `|`), `description` (склейка описаний), `element_names` (оригинальные имена для отображения), `n_elements`.

## 7. Группировка элементов (специфика нашего корпуса)

В `Показатели.csv` каждая строка — один показатель (элемент) публикации. При прямой индексации каждый элемент стал бы отдельным документом — на 1.7M элементов это 1.7M строк в индексе.

С `--group-size N` подряд идущие элементы одной публикации склеиваются в **группу** — один документ-«мега-элемент» с N склеенными `name` и `description`. На текущих данных с N=5: 16 068 элементов → 3 321 группа (×4.5 экономия). На прод-масштабе это будет ×N экономии по числу документов в индексе.

Это **асимметричная схема**: индекс строится по группам, но **запрос** в [coverage.py](../coverage.py) всё ещё идёт от ОДНОГО атрибута новой публикации. То есть «атрибут новой нашёл группу старой → +1 голос её публикации».

Группа выглядит для BM25 как обычный документ — просто длиннее (avgdl растёт с 17.5 до 59.9). Параметр `b` нормализует длину, относительный порядок документов в выдаче сохраняется.

## 8. Структуры в RAM при работе

При вызове [open_searcher](../bm25/searcher.py) грузится:

| Структура | Размер для 3 606 docs | Прогноз для 1.7M docs |
|---|---|---|
| `BM25Index.vocab` | ~150 КБ | ~5–15 МБ |
| `BM25Index.tf` (CSR) + позже tf_csc | ~1 МБ | ~30–60 МБ |
| `BM25Index.doc_lens` | 14 КБ | 7 МБ |
| `BM25Index.df` | 18 КБ | 1 МБ |
| `BM25Index.doc_ids` | ~100 КБ | ~50 МБ |
| `BM25Index.kinds`, `pub_ids` | ~100 КБ | ~50 МБ |
| `LazyDocuments._id_to_row` (dict) | ~300 КБ | ~150 МБ |
| `LazyDocuments._offsets` | 28 КБ | 14 МБ |
| `Tokenizer.morph` (pymorphy3) | ~30 МБ | ~30 МБ (одинаково) |

**Что НЕ грузится в RAM:** сами документы из `docs.jsonl`. Они читаются с диска точечно через `LazyDocuments.__getitem__` — `seek` по `offsets[row]`, `readline`, `json.loads`. На запрос обычно нужны метаданные 5–10 документов (топ из выдачи), это ~0.5 мс дисковой latency.

## 9. Сборка индекса (что происходит в `rebuild_and_save`)

[bm25/searcher.py:rebuild_and_save](../bm25/searcher.py) → [bm25/builder.py:build_streaming_from_jsonl](../bm25/builder.py).

Один проход по `docs.jsonl`:

```
для каждой строки jsonl:
   1. f.tell() → запомнить offset
   2. json.loads(line) → docПЕРЕД READLINE
   3. tokens = tokenizer(склейка полей по INDEX_FIELDS_BY_KIND[doc.kind])
   4. counts = Counter(tokens)
   5. для каждой пары (term, cnt) в counts:
        если term нет в vocab → добавить (новый col_index = len(vocab))
        добавить (row_index, col_index, cnt) в три параллельных списка rows/cols/vals
   6. doc_ids.append(doc.doc_id)
      kinds.append(doc.kind)
      pub_ids.append(doc.pub_id)
      doc_lens.append(len(tokens))
   7. row_index += 1

в конце:
   tf = scipy.sparse.csr_matrix((vals, (rows, cols)), shape=(N, V))
   df = (tf > 0).sum(axis=0)
   save_index(...) → 9 файлов в data/index/
```

RAM-профиль: vocab + три списка (rows, cols, vals) для COO + параллельные массивы метаданных. Никогда не держим в памяти больше одного документа за раз.

Время на текущих данных: ~1.6 с для 3 606 документов. Прогноз для 1.7M: 3–5 минут.

## 10. Поиск (что происходит в `Searcher.search`)

[bm25/searcher.py:Searcher.search](../bm25/searcher.py) + [bm25/index.py:score](../bm25/index.py).

```
1. q_tokens = tokenizer(query)
   "инфляционные ожидания" → ["инфляционный", "ожидание"]

2. term_ids = [vocab[t] for t in q_tokens if t in vocab]
   → [42, 17]  (пропускаем OOV)

3. для каждого term_id в term_ids:
   col = tf_csc[:, term_id]            ← разреженный столбец
   rows = col.indices                   ← документы, где термин встречается
   tf_vals = col.data                   ← их tf(t, d)

   norm = 1 − b + b · doc_lens[rows] / avgdl
   denom = tf_vals + k1 · norm
   contrib = idf[term_id] · tf_vals · (k1 + 1) / denom

   scores[rows] += contrib

4. (опционально) применить маску по kind / другим фильтрам:
   scores = np.where(mask, scores, 0.0)

5. top-K через argpartition:
   top_idx = argpartition(-scores, k)[:k]
   top_idx = top_idx[argsort(-scores[top_idx])]

6. для каждого row в top_idx:
   doc_id = index.doc_ids[row]
   doc = LazyDocuments[doc_id]  ← seek + readline + json.loads
   ← здесь происходит чтение с диска

7. вернуть [(scores[row], doc), ...]
```

Время на запрос: миллисекунды. Lazy-доступ к документам добавляет ~0.5 мс на каждый возвращаемый документ (на топ-5 — 2.5 мс).

## 11. Coverage поверх индекса

[coverage.py](../coverage.py) использует индекс необычно: вместо `Searcher.search` зовёт `bm25_score()` напрямую и сам делает фильтрацию по маске. Это потому что:
- Маска ковеража динамическая (исключение конкретного `pub_id`)
- Голосование по `top-K` с порогом и пропорциональным распределением — не та логика, что в обычном поиске

Алгоритм coverage:
```
для каждого атрибута target:
   scores = bm25_score(index, tokenize(attr.name + attr.description))
   scores = np.where(candidate_mask, scores, 0.0)  ← обнуляем чужие документы

   above_thr = scores > threshold
   top-K из above_thr
   weight = score / sum(top_K_scores)               ← пропорционально, сумма = 1.0
   для каждого row в top_K:
       pub_votes[index.pub_ids[row]] += weight

ранжируем pub_votes по убыванию → результат
```

При `--workers > 1` используется `multiprocessing.Pool` с `fork()`: индекс шарится между процессами через CoW. Каждый воркер получает один атрибут (уже токенизированный в main-процессе), считает голоса, возвращает dict. Main мерджит результаты.

## 12. Часто задаваемые вопросы себе

**Почему vocab — это dict, а не numpy-массив?**
Потому что ключи — строки переменной длины. numpy не умеет в строки эффективно. Поиск `vocab[term]` — это O(1) hashmap, отлично подходит для запросов.

**Почему IDF не сохраняется?**
Чтобы автоматически быть консистентным с актуальными `df` и `N`. Пересчёт занимает миллисекунды, экономия диска копеечная.

**Почему `doc_ids` отдельный файл, а не часть `meta.json`?**
`meta.json` — диагностика для людей, в неё можно открыть глазом. `doc_ids` может быть большой (десятки МБ при 1.7M документов), его открывать редактором не надо.

**Почему JSONL, а не БД?**
Намеренно. Корпоративное требование «жить в файликах», и для нашего масштаба разница в скорости несущественна. Плюс: append-only логика тривиальна, диффится в git, читается глазами.

**Почему `kinds` и `pub_ids` хранятся И в индексе, И в JSONL?**
В индексе — это плоские параллельные массивы для быстрых масок (`numpy.where`). В JSONL — это часть документа (источник правды). При rebuild индекса они переписываются из JSONL — никакого риска рассинхрона.

**Что произойдёт, если запустить `add_doc.py` на одном и том же CSV дважды?**
Дедупликация по `doc_id`: повторно добавляемые записи отфильтровываются, `docs.jsonl` не растёт, индекс не пересобирается. Скрипт идемпотентен.

**Можно ли обновлять документы?**
Прямо сейчас — нет. Это намеренное ограничение: источник правды append-only. Если документ изменился, нужно явно: удалить старую строку из jsonl + добавить новую + пересобрать. Реальный кейс пока не возникал, и архитектура не оптимизирована под обновления.
